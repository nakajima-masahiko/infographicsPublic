# リリースポリシー

本ドキュメントは、このリポジトリにおけるバージョニング、互換性、リリースノート運用を統一するための基準です。

## 1. SemVer採用方針（MAJOR / MINOR / PATCH）

本プロジェクトは **Semantic Versioning（SemVer）** を採用します。

- **MAJOR（x.0.0）**
  - 後方互換性のない変更を含む場合に更新する。
  - 例: 既存の公開API/設定キー/出力仕様の削除・変更。
- **MINOR（0.x.0 / x.y.0）**
  - 後方互換性を維持した機能追加を行う場合に更新する。
  - 例: 新機能、新オプション、新しいテンプレート追加。
- **PATCH（0.0.x / x.y.z）**
  - 後方互換性を維持した不具合修正や軽微改善の場合に更新する。
  - 例: バグ修正、ドキュメント補足、性能改善（仕様変更なし）。

## 2. β期間中の互換性ルール（破壊的変更の扱い）

β期間（`0.y.z`）では以下を適用します。

- `0.y.z` の **MINOR（y）更新時** は、破壊的変更を含めることを許容する。
- ただし破壊的変更を行う場合は、必ず以下を実施する。
  - `CHANGELOG.md` の **[Unreleased] > Breaking Changes** へ明記する。
  - 影響範囲（対象機能、移行手順、代替手段）をリリースノートに記載する。
  - 必要に応じて旧仕様の廃止予定（deprecation）を事前告知する。
- `0.y.z` の **PATCH（z）更新** では、破壊的変更を禁止する。

## 3. リリースノート作成手順

リリースノートは `CHANGELOG.md` を正本として作成します。

1. 開発中は `## [Unreleased]` セクションを更新する。
2. 変更内容は以下の見出しに分類する。
   - `### Added`（追加）
   - `### Changed`（変更）
   - `### Fixed`（修正）
   - `### Breaking Changes`（破壊的変更）
3. リリース時に `Unreleased` の内容を新しいバージョン見出しへ移動する。
   - 例: `## [0.3.0] - 2026-03-11`
4. 同時に新しい空の `Unreleased` セクションを先頭に再作成する。

## 4. タグ命名規則

Gitタグは以下で統一します。

- 形式: `vMAJOR.MINOR.PATCH`
- β期間の例: `v0.x.y`（例: `v0.3.1`）
- 安定版の例: `v1.0.0`

補足:
- タグ名と `CHANGELOG.md` のバージョン見出しを一致させる。
- 公開日時は `CHANGELOG.md` の日付（`YYYY-MM-DD`）で管理する。

## 5. 公開前チェック

公開前は、最低でも以下を確認すること。

- バージョン番号が SemVer ルールに従っている。
- `CHANGELOG.md` の `Unreleased` がリリース見出しへ反映済みである。
- `Breaking Changes` の有無と移行手順が明記されている。
- タグ名が `vMAJOR.MINOR.PATCH` 形式になっている。
- 公開対象（成果物・ドキュメント）とリリースノート内容が一致している。

運用上、上記確認は PR の最終確認項目として扱い、マージ前に必ず完了させること。
