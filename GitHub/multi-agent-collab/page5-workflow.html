協調ワークフロー 実践

Issue から PR まで全自動のマルチエージェントワークフロー。
Spec-Driven 開発・ガードレール設計・ハンドオフのベストプラクティス。

標準マルチエージェント開発フロー
1
要件定義・Issue 作成 人間
プロダクトオーナーまたは開発者が GitHub Issue に要件を記述する。ユーザーストーリー・受け入れ条件・関連 Issue リンクを含める。
📋 GitHub Issue (要件)
2
仕様策定・タスク分解
PM / Spec Agent
PM エージェントが Issue を読み込み、spec.md に詳細仕様を作成。タスクを tasks.md に分解し、各エージェントへのアサインを準備する。
📄 spec.md → plan.md → tasks.md
3
仕様レビュー・承認 人間承認
Tech Lead (人間)
技術リードが spec.md をレビューし承認。スコープ・アーキテクチャ方針・リスクを確認。重要なチェックポイントで人間の判断を挟む。
✅ 承認済み spec.md
4
並列実装・テスト作成
Developer Agent × NTester AgentDocumentation Agent
複数の Developer エージェントが git worktree で並列実装。Tester エージェントが同時にテストを作成。Documentation エージェントがドキュメントを更新。
🌿 Feature branches × N
5
セキュリティ・アーキテクチャチェック
Security AgentArchitecture Guardian
Security エージェントが脆弱性・シークレット漏洩・依存 CVE をスキャン。Architecture Guardian が設計原則への適合を確認。問題があれば Developer に差し戻し。
🔒 セキュリティレポート
6
コードレビュー・PR 作成
Code Reviewer Agent
Reviewer エージェントが diff をプロジェクトコンテキストと共にレビュー。インラインコメントを投稿し、問題があれば Developer エージェントに修正を委託。クリアなら PR をマージ準備状態に。
📬 Draft PR → Ready for Review
7
最終承認・マージ・デプロイ
DevOps AgentTech Lead (人間) 承認
人間が最終 PR をレビューし承認。DevOps エージェントがCI/CD を実行し本番へデプロイ。リリースノートも自動生成。
🚀 本番デプロイ完了

Spec-Driven の利点

全エージェントが同じ spec.md を参照することで、スコープ逸脱・ハルシネーション・矛盾した実装を防ぎます。仕様が変わった場合も spec.md を更新するだけで全エージェントが追従できます。「仕様を書く時間」が最大の投資対効果をもたらします。

ガードレール設計 — 安全な自律実行のために

絶対禁止（ハードガードレール）

main ブランチへの直接プッシュ禁止 / 本番 DB への直接書き込み禁止 / 認証情報のコミット禁止 / 他エージェントのワークツリーへの書き込み禁止

人間承認必須（ソフトガードレール）

本番デプロイ / 外部 API への書き込み / セキュリティポリシー変更 / 依存関係のメジャーバージョンアップ / コスト100ドル以上の操作

自律実行 OK（許可済みアクション）

feature ブランチへのコミット / テストの実行・修正 / ドキュメントの更新 / ドラフト PR の作成 / セキュリティスキャンの実行・レポート

コスト管理（トークン制限）

エージェントごとの最大トークン数設定 / タスクの最大実行時間制限 / 無限ループ検出・中断機構 / 定期的な進捗確認ポイント設定

失敗パターンと対策

最も多い失敗は「タスク境界の曖昧さ」です。2つのエージェントが同じファイルを同時に編集しようとする競合、スコープを超えた変更、引き継ぎ情報の欠落が主な原因です。対策は「git worktree で作業空間を分離」「アーティファクト（ファイル）を介したハンドオフ」「明示的なタスク所有権の宣言」の3点です。
