Spec-Driven の次に来る Intent-Driven 開発

仕様は正しさを作るが、意図は方向を作る。AIが強くなるほど、上流の曖昧さは実装リスクへ直結する。
そのため spec.md の前に intent.md を置く設計が、次の3年の実務で重要になる可能性がある。

spec-driven の限界
  • 仕様通りに実装できても、なぜ作るかが弱いと成果は分散する。
  • 優先順位が曖昧なままだと、AIは局所最適を量産しやすい。
  • 禁止事項や美意識が未定義だと、品質が「正しいが違和感がある」状態になる。
intent.md の役割：方向性を固定する文書

なぜ作るのか

課題背景と制度上の目的を定義する。

誰の痛みを解くか

対象ユーザーと痛みの優先順位を明確化。

何をやらないか

意図的に捨てる要素を明文化して暴走を防ぐ。

価値の優先順位

速度・信頼性・体験・コストなどの序列を明示。

守る体験や思想

プロダクトの一貫性を維持する設計原則を固定。

比較：Issue-Driven / Spec-Driven / Intent-Driven
方式強み弱み適した局面
Issue-Driven着手が速い方向性の統制が弱い小修正・短期対応
Spec-Driven正確性と再現性意図の濃度が不足しがち中規模以上の実装
Intent-Driven構造変化に強い意思決定上流設計の負荷が高いAI協働・長期プロダクト戦略

結論

AI時代の上流工程は削減されるのではなく、むしろ濃くなる。人間の役割は仕様を書く人から、意図を定義し続ける人へシフトする。意図が弱い組織ほど、実装速度が上がっても方向を見失うリスクが上がる。
