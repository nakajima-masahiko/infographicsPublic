AIエージェントの役割分担 設計

PM・開発・レビュー・テスト・セキュリティ・ドキュメント・アーキテクチャ・DevOps。
8つの役割と責務、使用ツール、連携方法を完全解説。

8つのエージェント役割
📋

PM / Spec エージェント

プロジェクト管理・仕様策定

要件を仕様ドキュメントに変換し、タスクに分解してチームに割り当てる。spec.md → plan.md → tasks.md のパイプラインを管理し、仕様が進化するにつれて更新する。

主な責務

  • 要件をspec.md に構造化
  • タスクを tasks.md に分解・優先度付け
  • Issue 作成・ラベル付け・アサイン
  • 仕様変更の影響範囲を特定
  • 進捗の追跡とレポート作成
GitHub Issues API Projects spec.md
💻

Developer / Feature Author エージェント

コーディング・実装

タスク仕様から実際の実装コードを作成する。ファイルの作成・編集・ビルドコマンドの実行・デバッグを行い、完成したらドラフトPRを作成する。

主な責務

  • tasks.md のタスクを実装コードに変換
  • 複数ファイルにわたる変更を管理
  • ビルド・リント・単体テストの実行
  • エラーの自己修正・デバッグ
  • ドラフトPRの作成とコメント
ファイル読み書き シェル実行 Git
👁️

Code Reviewer エージェント

コードレビュー・品質管理

PR の差分全体をプロジェクトコンテキストと共に確認し、スタイル・ロジック・アーキテクチャ適合性をレビュー。インラインコメントを投稿し、修正PRを自動生成することもできる。

主な責務

  • コーディング規約への適合確認
  • ロジックのバグ・エッジケース検出
  • アーキテクチャ原則の違反を指摘
  • PR インラインコメント投稿
  • 修正提案を Developer に委託
GitHub PR API Diff 解析 コード検索
🧪

Tester / QA エージェント

テスト作成・品質保証

ユニットテスト・統合テストを生成し、テストスイートを実行する。失敗を分析し、カバレッジギャップをレポートする。TDD 強制にも対応。

主な責務

  • ユニット・統合テストの自動生成
  • テストスイートの実行・分析
  • テストカバレッジのレポート
  • 失敗したテストの原因分析
  • エッジケース・境界値テストの設計
テストフレームワーク カバレッジツール CI
🛡️

Security エージェント

セキュリティ・コンプライアンス

コードの脆弱性をスキャンし、異常なアクセスパターンを監視する。OWASP Top 10・依存関係の CVE・コンプライアンスポリシーを PR ごとにチェック。

主な責務

  • OWASP Top 10 脆弱性スキャン
  • 依存関係の CVE チェック
  • シークレット・認証情報の漏洩検出
  • コンプライアンスポリシーの強制
  • セキュリティレポートの作成
SAST ツール Dependabot Secret scan
🏛️

Architecture Guardian エージェント

アーキテクチャ整合性管理

新しいコードがアーキテクチャパターンと設計決定に準拠しているかを確認。設計原則から逸脱したコードをブロックし、技術的負債の蓄積を防ぐ。

主な責務

  • 設計パターンへの準拠確認
  • レイヤー境界違反の検出
  • API 設計の一貫性チェック
  • 技術的負債のスコアリング
  • ADR（アーキテクチャ決定記録）の更新
ADR ドキュメント コード解析 依存グラフ
📚

Documentation エージェント

ドキュメント作成・維持管理

API ドキュメント・インラインコメント・ナレッジベースを自律的に作成・更新する。コード変更に追従してドキュメントが常に最新状態を維持するよう管理する。

主な責務

  • API 仕様書（OpenAPI等）の自動生成
  • コードコメント・JSDoc の追加
  • README・Wiki の更新
  • 変更履歴・リリースノートの作成
  • ドキュメントとコードの乖離検出
OpenAPI Wiki API Markdown
🚀

DevOps / SRE エージェント

CI/CD・インフラ・信頼性

CI/CD パイプライン設定・Infrastructure as Code・デプロイメント・信頼性チェックを管理する。ビルド失敗の自動診断とパイプライン最適化を行う。

主な責務

  • GitHub Actions ワークフロー管理
  • Terraform/Helm 等の IaC 更新
  • デプロイメントパイプラインの実行
  • ビルド失敗の自動診断・修正
  • SLO/SLA 監視レポートの作成
GitHub Actions Terraform クラウドAPI
役割設計の原則
01
単一責任原則
1つのエージェントは1つの役割のみ。複数の役割を持たせると精度が下がり、障害の特定が困難になる。
02
最小権限アクセス
各エージェントは業務に必要なツール・ファイルのみにアクセス可能にする。セキュリティと予期せぬ副作用を防ぐ。
03
明確なハンドオフ
エージェント間の引き継ぎには構造化されたアーティファクト（PR・ドキュメント・ラベル）を使用する。暗黙の前提を排除。
04
人間のチェックポイント
重要な決定（仕様承認・本番デプロイ・セキュリティ修正）には必ず人間の承認ステップを設ける。完全自動化は慎重に。

Stripe Minions からの示唆

Stripe は隔離された VM 上で動く自律エージェント群「Minions」を使い、人間が1行もコードを書かずに週 1,000〜1,300 PR 以上をマージしています。成功の鍵は「エージェントに単一のエンド・ツー・エンドタスクを割り当てること」と「隔離された実行環境」です。大きすぎる責務を1エージェントに持たせると失敗率が急上昇します。
