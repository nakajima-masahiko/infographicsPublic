Issue → PR 自動化フロー

タスク割り当てから Pull Request 作成・反復サイクルまでの全ステップ

🔄

8 ステップ実行フロー

1

📋 タスク割り当て

GitHub Issue を Copilot にアサイン、または Copilot Chat でタスクを依頼します。Issue のタイトル・説明・コメントすべてが Copilot へ渡されます。

割り当て後に Issue へ追加したコメントは Copilot には届きません。追加情報はPRコメントで行います。
2

🌿 ブランチ＆ドラフトPR 即時作成

Copilot は即座に copilot/* ブランチを作成し、空のコミットでドラフト Pull Request を開きます。作業の進捗はここに随時プッシュされます。

3

⚙️ 環境セットアップ

GitHub Actions のエフェメラルサンドボックスが起動し、リポジトリをクローン。copilot-setup-steps.yml が設定されていれば依存関係のインストールや言語ランタイムのセットアップを自動実行します。

4

🔍 コンテキスト分析

リポジトリ構造・ファイル内容・Issue の詳細・カスタム指示ファイルを読み込み、タスクのチェックリストを作成。PRの説明欄に計画を書き込みます。

読み込む情報: copilot-instructions.md / .github/instructions/ 内ファイル / Issue コメント
5

💻 実行ループ（コード・テスト・修正）

ファイル編集 → テスト実行 → リンター実行 → エラー確認 → 自動修正 のループを繰り返します。チェックリストの各項目を順番に完了させ、コミットをプッシュしながら進めます。

6

🛡️ セキュリティスキャン

GitHub の Secret Protection・Code Security・Supply Chain Security ツールを使い、作成したコードに脆弱性・シークレットの漏洩がないかチェックします。問題があれば修正してから次へ。

7

🔬 自己レビュー

Copilot Code Review を自分の PR に適用し、フィードバックを取り込んで品質を高めます。人間にレビューを依頼する前に可能な限り自己改善します。

8

👤 人間レビュー依頼＆反復

ドラフト PR を正式にオープンし、担当開発者へレビューを依頼します。PR コメントで @copilot をタグ付けしてフィードバックを伝えると、Copilot が修正を反復します。

Copilot は自分の PR を承認・マージできません。また、タスクを依頼した本人も承認者になれません（必ず独立したレビューが必要）。

📦

コミットの透明性

Copilot が作成するすべてのコミットには以下の情報が含まれ、監査・コードレビューを容易にします。

feat: Implement dark mode toggle for settings panel

Co-authored-by: yamada-taro <yamada@example.com>
Agent session: https://github.com/org/repo/actions/runs/xxxxx

コミットの著者: Copilot（自動生成） / 共著者: タスクを依頼した開発者
コミットメッセージにはエージェントセッションログへのリンクが含まれ、実行内容を詳細に確認できます。

重要: Copilot が作成した PR の CI/CD ワークフローは、書き込み権限を持つユーザーが承認するまで自動実行されません。 これはセキュリティ上の保護であり、権限昇格攻撃を防ぎます。
