Claude Code の「記憶と指示設計」
実務で迷わない1枚図

ポイントは「どこに書くか」「どう分割するか」「どう衝突を防ぐか」。
以下は Anthropic公式ドキュメントに基づく、運用可能な最小構成です。

2系統 CLAUDE.md と Auto memory
4スコープ Managed / Project / User / Subdir
~200行目安 1ファイルを短く保つ

① まず全体像：Claudeの記憶は2本立て

CLAUDE.md：人間が書く恒久ルール（コーディング規約/作業手順/設計方針）
Auto memory：Claudeが学習した補助メモ（修正履歴や好み）

実務メモ：重要ルールは必ず CLAUDE.md 側へ。Auto memory任せにしない。

② どこに置く？優先順位の考え方

  • /etc/claude-code/CLAUDE.md など Managed（組織共通）
  • ./CLAUDE.md or ./.claude/CLAUDE.md（プロジェクト共通）
  • ~/.claude/CLAUDE.md（個人共通）
  • サブディレクトリ内 CLAUDE.md（該当ファイルを読む時に読み込み）

注意：広いルールほど抽象、狭いルールほど具体にして衝突を防ぐ。

③ 分割の正解：肥大化したら import / rules

1ファイルが長くなったら、@path/to/file で分割。さらにテーマ別は .claude/rules/ を使う。

# CLAUDE.md 例
@AGENTS.md
@docs/git-instructions.md

## Claude Code専用
- 変更前に影響範囲を箇条書き
- 生成後に必ずテストコマンドを実行

公式準拠：既存の AGENTS.md 文化があるリポジトリは、CLAUDE.md で import 連携できる。

④ 初期セットアップの最短ルート

/init でプロジェクト初期の CLAUDE.md を生成
内容を200行以内目安で整理（冗長な精神論を削る）
命令文は検証可能に書く（例: 「コミット前に npm test」）

曖昧ルールより、検証できるルールの方が遵守率が高い。

⑤ サブエージェント運用（必要な時だけ）

  • .claude/agents/（プロジェクト）と ~/.claude/agents/（個人）
  • YAML frontmatterで name / description / tools を定義
  • 責務は小さく分ける（reviewer, debugger, docs-writer など）

設計原則：1エージェント1責務 + 最小権限ツール。

⑥ そのまま使える運用テンプレ

CLAUDE.md
1. プロジェクト概要（3行）
2. 絶対ルール（5〜10項目）
3. 開発フロー（編集→テスト→コミット）
4. ディレクトリ規約
5. import（詳細は別ファイルへ）

.claude/rules/
- code-style.md
- testing.md
- security.md

この構成だと、チーム拡大時もルール追加が壊れにくい。
