結論①

エージェント開発のボトルネックは「モデル能力不足」より、ハーネス（制約・文脈・検証）不足である。

結論②

ハーネス改善だけで、同一モデルでも性能は大きく伸びうる（LangChainの実証）。

結論③

実運用では「初期構築」より「継続メンテ（ドキュメント更新・ガーベジコレクション）」が勝負を分ける。

1. 原典から見る“ハーネス”の定義

OpenAI（Ryan Lopopolo, 2026-02-11）の実験は、 人間が1行もコードを書かない制約下で、 5ヶ月・約100万行規模を運用可能にした事例です。 そこで示された中核は「Humans steer. Agents execute.」という役割分担でした。

つまり、エンジニアの仕事は「コードを書くこと」から、 「エージェントが正しく書ける環境（ハーネス）を設計すること」へシフトします。 ハーネスは単一のプロンプトではなく、文脈設計・制約設計・検証ループを束ねた運用システムです。

Context

AGENTS.mdを目次化し、真の知識はdocs配下の構造化ドキュメントに分散。

Constraints

カスタムリンター・構造テストで依存方向や設計規約を機械的に強制。

Continuous Cleanup

定期的な自動リファクタPRでエントロピーを返済（ガーベジコレクション）。

2. 批評から学ぶ：なぜ“効く”のか、どこが難しいのか

Martin Fowler系の論点（2026-02-17）

  • OpenAI事例を、①Context Engineering ②Architectural Constraints ③Garbage Collection の3軸で整理。
  • 将来は「ハーネス＝サービステンプレート化」する可能性を示唆。
  • 同時に、既存巨大コードベースへの適用や機能検証の厳密化は未解決課題として指摘。

現場で詰まりやすいポイント

  • 制約が弱い：速いが壊れる。
  • 制約が強すぎる：安全だが進まない。
  • ドキュメントが古い：エージェントが過去の前提で誤実装する。
  • 評価不足：見かけの成功に対し、本番要件を満たしていない。

3. 実証：モデル固定でも性能は上がる（LangChain）

Terminal Bench 2.0 で 52.8 → 66.5（Top 30 → Top 5）

変更したのはモデルではなく、ハーネス（自己検証、環境コンテキスト注入、ループ検出、推論配分）。

施策 実装例 効果
自己検証ループ 終了前にPreCompletionChecklistでテスト実行を強制 「書いて終わり」を減らし正答率改善
環境コンテキスト注入 起動時にディレクトリとツールを探索・注入 探索ミス減、初動高速化
ループ検出 同一ファイル編集回数を監視し警告 doom loop抑制
推論予算配分 xhigh-high-xhigh の reasoning sandwich 品質と時間制限のバランス最適化

4. 90日導入プレイブック（実務向け）

Day 1-14

  • AGENTS.mdを100行前後に圧縮（目次化）
  • docs配下に設計判断の一次情報を移設
  • 最低限のlint / test / CIゲートを整備

Day 15-45

  • カスタムリンター導入（依存方向・命名・ログ）
  • エージェントの自己検証チェックリスト導入
  • 失敗トレース収集と失敗分類を開始

Day 46-90

  • 定期ガーベジコレクションPRを自動化
  • ドキュメント鮮度チェックをCIへ組み込み
  • モデル差し替え耐性（provider-agnostic）を検証

5. よくある誤解

誤解：良いモデルを選べば解決する。
実際：同一モデルでもハーネス差で成績が大きく変わる。

誤解：AGENTS.mdを長くすれば賢くなる。
実際：巨大単一ファイルは劣化しやすく、重要情報が埋もれる。

誤解：一度作れば完成。
実際：ハーネスはプロダクト同様に継続改善が必要。
